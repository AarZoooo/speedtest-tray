# macOS Porting Progress & Architecture Context

This document captures the implementation details, design decisions, and progress of porting SpeedTest Tray to macOS. It serves as a handoff context for any agent/developer picking up work on this repository.

---

## 1. Goal Overview
The goal is to enable the Go/Wails v2 application (`speedtest-tray`) to run seamlessly on macOS as a native menu bar status application (accessory app). 
- It must stay hidden from the macOS Dock.
- It must display a template menu bar icon matching system theme modes (light/dark).
- Left-clicking the status icon toggles the webview window, centered directly beneath the icon.
- Right-clicking (or Ctrl+clicking) the status icon presents a native context menu (Show, Enable Session Logging, Quit).
- Quitting the application cleanly cleans up resources and immediately removes the status item from the macOS menu bar.

## 2. Architecture & File Layout

To avoid compiling Windows-only dependencies (`github.com/energye/systray`) on macOS and vice versa, we segregated platform logic using Go build tags.

### Platform-Specific File Layout
- **[main.go](file:///Users/devjotikundu/Documents/speedtest-tray/main.go)**: Entrypoint. Calls platform-neutral `gui_wails.StartTray(...)` and registers lifecycle hooks (`OnStartup: app.Startup`, `OnShutdown: app.Shutdown`).
- **[internal/gui_wails/app.go](file:///Users/devjotikundu/Documents/speedtest-tray/internal/gui_wails/app.go)**: Tracks window visibility state (`windowVisible`) and bridges lifecycle events (`Startup` -> `initMacStatusItem`, `Shutdown` -> `cleanupMacStatusItem`).
- **[internal/gui_wails/tray_darwin.go](file:///Users/devjotikundu/Documents/speedtest-tray/internal/gui_wails/tray_darwin.go)** (`//go:build darwin`): CGO bridge wrapper. Exposes callback exports (`onStatusItemClick`, `onQuitClick`, `onToggleLoggingClick`) and interfaces with the Objective-C runtime.
- **[internal/gui_wails/tray_darwin.m](file:///Users/devjotikundu/Documents/speedtest-tray/internal/gui_wails/tray_darwin.m)**: Native AppKit Objective-C implementation. Spawns `NSStatusItem`, manages the status bar button actions, handles click routing (left vs. right click), populates the context menu, and translates screen coordinates.
- **[internal/gui_wails/window_darwin.go](file:///Users/devjotikundu/Documents/speedtest-tray/internal/gui_wails/window_darwin.go)** (`//go:build darwin`): macOS-specific window positioning logic. Fetches the status item's screen coordinate frame, converts AppKit's bottom-up Y-coordinates to Wails' top-down coordinate space, and aligns/centers the window.
- **[internal/gui_wails/tray_windows.go](file:///Users/devjotikundu/Documents/speedtest-tray/internal/gui_wails/tray_windows.go)** (`//go:build windows`): Windows-only tray implementation using `energye/systray`.
- **[internal/gui_wails/tray_other.go](file:///Users/devjotikundu/Documents/speedtest-tray/internal/gui_wails/tray_other.go)** (`//go:build !windows && !darwin`): No-op fallback for compilation compatibility on other platforms (e.g., Linux).
- **[internal/gui_wails/window_other.go](file:///Users/devjotikundu/Documents/speedtest-tray/internal/gui_wails/window_other.go)** (`//go:build !windows && !darwin`): No-op fallback for window positioning on non-macOS/non-Windows platforms.

## 3. Key Issues Resolved In This Session

### A. App Bundle Reporting as "Damaged or Incomplete"
- **Cause**: By default, the `Info.plist` template generated by Wails populated `CFBundleExecutable` with the project name `{{.Name}}` (which interpolates to `SpeedTest Tray`), while the binary built in production is named `speedtest-tray-portable` (configured in `wails.json`). macOS rejected the bundle because the executable name did not match the actual binary.
- **Fix**: Explicitly set `<key>CFBundleExecutable</key><string>speedtest-tray-portable</string>` inside `build/darwin/Info.plist`.

### B. Ghost Menu Bar Icon on Quit
- **Cause**: Clicking "Quit" on the menu bar initiated the shutdown via Go's `globalApp.Quit()`, but because the Wails `OnShutdown` option was not configured in `main.go`, `app.Shutdown()` (which calls CGO status item cleanup) was never invoked. The status icon would stay on the menu bar until clicked again.
- **Fix**: Configured `OnShutdown: app.Shutdown` in `newOptions` inside `main.go` and implemented cross-platform no-ops for `initMacStatusItem` / `cleanupMacStatusItem` to make sure Windows and Linux targets still compile.

### C. Visual Aesthetics (Sharp Behind Rounded Webview)
- **Cause**: macOS renders a rectangular translucent vibrancy container backing for Wails windows when `WindowIsTranslucent` is enabled, which peeked out behind the frontend's CSS rounded corners.
- **Fix**: Set `WindowIsTranslucent: false` in `main.go` for `mac.Options`. This leaves the window backing completely transparent, displaying only the CSS-rounded webview container.

### D. Blurry or Overly Bold Typography on macOS
- **Cause**: Browser rendering differences between Windows and macOS caused text sizes/weights to look inconsistent.
- **Fix**: Added `-webkit-font-smoothing: antialiased;` and `-moz-osx-font-smoothing: grayscale;` to `body` in `frontend/style.css`.

### E. Main Thread Blocking on Click Callbacks
- **Cause**: CGO calls into Go callbacks are executed synchronously. Performing window actions (like showing or hiding) directly inside these callback routines blocked the OS/Objective-C main event loop.
- **Fix**: Spawned all window and state updates asynchronously from Go (`go globalApp.ToggleWindow()`).

## 4. Verification & Testing

### Automated Tests
Verify backend unit tests and frontend vitest test suites before building:
```bash
# Backend tests
go test ./...

# Frontend tests
npm test --prefix frontend
```

### Building the Production Bundle
To build the native macOS app bundle:
```bash
# Build the .app bundle using the Wails CLI
~/go/bin/wails build

# Copy the built app to your Desktop for quick verification
rm -rf ~/Desktop/SpeedTest\ Tray.app && cp -R build/bin/SpeedTest\ Tray.app ~/Desktop/
```
